Thought for 15s
You want a portable Rust control plane for interactive AI-agent terminals on macOS and Linux, where:

each agent runs inside a PTY, not a normal inherited stdin/stdout shell,

your coordinator can inject keystrokes/commands into that PTY when needed,

the agent still behaves like it is in a real terminal,

you can optionally share control with a human operator,

and you want to reduce manual “poke this agent now” work.

The cleanest architecture is:

High-level design
Core idea
Do not try to “share the keyboard” of an already-open Terminal/iTerm/GNOME Terminal tab.

Instead, make your Rust program own the PTY master for every agent session, and run the agent process on the PTY slave. Then your program becomes the terminal multiplexer / broker:

reads agent output from PTY master

writes operator input to PTY master

writes coordinator-triggered commands to PTY master

records transcript/state

detects prompts / idleness / blockers

injects reminders like “read the coordination wiki page X”

That is the right abstraction boundary.

On Unix-like systems, PTYs are exactly for this: openpty() gives you a master/slave pair, and forkpty() is the classic “spawn a child attached to a pseudo-terminal” helper. On macOS specifically, forkpty() is documented as combining openpty(), fork(), and login_tty(). 

Best Rust crate choice
For macOS + Linux portability, the first thing I would reach for is portable-pty. Its docs explicitly describe it as a cross-platform PTY API, and it supports opening a PTY, spawning a shell/command into it, cloning a reader, and writing to the PTY master. It is part of the WezTerm ecosystem. 

That is better than building directly on nix::pty unless you specifically want low-level Unix-only control.

Keystroke injection model
Do not literally model every action as a physical keypress unless you need that.

Use three levels:

Shared keyboard model
You said you want “some sort of pty mechanism that lets me share the keyboard with the tool so we can both type commands.”

That is feasible, but you should make it serialized, not true concurrent typing.

Recommended arbitration
For each agent PTY, allow exactly one active writer at a time:

Human

Coordinator

Macro

None

Then support:

request control

grant control

preempt for emergency

release control

Why? Because if human and coordinator both write at the same time, the bytes interleave and you get garbage.

So “shared keyboard” should really mean:

both are authorized writers,

but writes are queued and serialized.

Practical rule
human interactive attach = exclusive writer

coordinator can enqueue “pending poke”

when human detaches or prompt is idle, coordinator injects pending command

That will feel much saner.

Detecting when it is safe to inject
This is important.

If you inject text while the agent is mid-command, you may corrupt its input stream.

Use some combination of:

1. Prompt detection
Recognize shell prompt patterns:

custom PS1

special sentinel prompt like __AGENT_READY__>

Best practice:
launch each shell with a known prompt.

For example, start bash/zsh with startup config that sets:

export PS1="__AGENT__$ "
Then your parser can detect readiness reliably.

2. Bracket commands with sentinels
When you inject commands, wrap them:

printf '__CMD_START__\n'
some-command-here
status=$?
printf '__CMD_DONE__:%s\n' "$status"
Now your supervisor can parse command completion robustly.

3. Idle timers
If no output for N seconds and prompt visible, agent is likely ready.

4. Optional shell helper
Install a tiny shell helper function in the agent session:

agent_read_wiki() { ... }
agent_ack() { ... }
Then coordinator commands become simple and structured.

Suggested control protocol
Instead of blindly typing arbitrary prose into agent terminals, give each agent a tiny local command vocabulary.

For example:

coord sync
coord inbox
coord open REQ-17
coord ack REQ-17
coord status
coord unblock
That means the coordinator only needs to inject:

coord inbox
and the agent-side tooling does the right thing.

This is much more reliable than pasting long English instructions every time.

Minimal code sketch
Here is the basic PTY pattern.

use anyhow::{Context, Result};
use portable_pty::{native_pty_system, CommandBuilder, PtySize, PtySystem};
use std::io::{Read, Write};
use std::sync::{Arc, Mutex};
use std::thread;

pub struct PtyAgent {
    writer: Arc<Mutex<Box<dyn Write + Send>>>,
}

impl PtyAgent {
    pub fn spawn(shell: &str, cwd: &std::path::Path) -> Result<Self> {
        let pty_system = native_pty_system();

        let pair = pty_system
            .openpty(PtySize {
                rows: 40,
                cols: 120,
                pixel_width: 0,
                pixel_height: 0,
            })
            .context("openpty failed")?;

        let mut cmd = CommandBuilder::new(shell);
        cmd.cwd(cwd);

        let _child = pair
            .slave
            .spawn_command(cmd)
            .context("spawn_command failed")?;

        let mut reader = pair
            .master
            .try_clone_reader()
            .context("clone reader failed")?;

        let writer = pair.master.take_writer().context("take writer failed")?;
        let writer = Arc::new(Mutex::new(writer));

        thread::spawn(move || {
            let mut buf = [0u8; 8192];
            loop {
                match reader.read(&mut buf) {
                    Ok(0) => break,
                    Ok(n) => {
                        let s = String::from_utf8_lossy(&buf[..n]);
                        print!("{s}");
                    }
                    Err(err) => {
                        eprintln!("pty read error: {err}");
                        break;
                    }
                }
            }
        });

        Ok(Self { writer })
    }

    pub fn send_line(&self, line: &str) -> Result<()> {
        let mut w = self.writer.lock().unwrap();
        w.write_all(line.as_bytes())?;
        w.write_all(b"\n")?;
        w.flush()?;
        Ok(())
    }

    pub fn send_raw(&self, bytes: &[u8]) -> Result<()> {
        let mut w = self.writer.lock().unwrap();
        w.write_all(bytes)?;
        w.flush()?;
        Ok(())
    }
}
That is the core mechanism: your program owns the PTY and can write commands to it at will.

Yes, this makes sense.

In fact, this is the right simplification:

PGM owns the terminals

PGM relays agent output to the browser

PGM relays your input back to the agent

PGM is the only router for inter-agent requests

wiki remains durable shared context

Ctrl-C + claude go remains a first-class workflow primitive

The only thing I would change is this:

Do not make wiki-reading the primary push protocol
Make the primary protocol:

direct PTY nudge when target is known

wiki update as durable context

optional broadcast “read coordination page” only when necessary

That will keep the system much more responsive and much less noisy.

Also, from the current official docs, Claude Code is clearly designed around interactive terminal usage, and the official docs now also expose an Agent SDK and remote-control-related capabilities, which reinforces the idea that a session-owning broker is a sensible direction even if you still start with PTYs first.

I generated the PoC Rust project as a downloadable zip:

pgm-poc.zip

It includes:

portable-pty-based PTY ownership

a single mode:

starts one shell in a PTY

cds to a runtime dir

runs a looping shell script

sends multiple Ctrl-C bytes

a duo mode:

starts two PTYs in two dirs

each runs a mock-agent shell script

agent A writes a JSON request

PGM detects it and injects the routed command into agent B’s PTY

agent B later writes a response JSON

PGM routes that back to agent A

logs PTY output and supervisor routing

sends Ctrl-C twice to each agent at shutdown

Cargo.toml

src/main.rs

README.md
